# Claude Code SDLC Harness

## Session Init

At session start, load in order:
1. Read `docs/SKILL_REGISTRY.md` ŌĆö skill index, phase order, ACTIVE_TASK section map
2. Read `ACTIVE_TASK.md` ŌĆö current task state
3. **Session health check** ŌĆö inspect ACTIVE_TASK.md state:
   - Empty ŌåÆ clean start, ready for `/task`
   - `## Requirement` populated but all other sections empty ŌåÆ task was started but not continued; resume with `design` or reset
   - `## Requirement` populated with some later sections also populated ŌåÆ find last section with an Observation block; resume from the next skill
   - Any section populated without an Observation block ŌåÆ that phase may be incomplete; warn user before proceeding
   - Non-empty ACTIVE_TASK with no Observation blocks anywhere ŌåÆ stale state from before observability was added; proceed but acknowledge
4. If `.claude/context/BE_CONTEXT.md` exists ŌåÆ read it (backend codebase snapshot)
5. If `.claude/context/FE_CONTEXT.md` exists ŌåÆ read it (frontend codebase snapshot)

Context files are generated by `close` after each merge. They are the fastest way to understand the codebase without re-scanning source files. Always prefer them over cold file reads when starting a session.

## AI-Native Engineering Principles

Seven principles embedded as first-class constraints in every skill:

| # | Principle | Harness Mechanism |
|---|-----------|-------------------|
| 1 | **Initialization needs its own phase** | `task` warns on non-empty ACTIVE_TASK; `init` checks for existing files; Phase 0 in all workflows |
| 2 | **Agents overreach and under-finish** | All skills hard-block on missing prior phase; `code`/`design`/`refactor` spawn `sdlc-investigator` |
| 3 | **Feature lists are harness primitives** | `code`, `tdd`, `verify` read acceptanceCriteria from `## Requirement` ŌĆö never from design or memory |
| 4 | **Agents declare victory too early** | `verify` requires test-runner evidence; `review` blocks on self-reported verdicts; Observation blocks gate all downstream |
| 5 | **End-to-end testing changes results** | `tests` requires Ōēź1 E2E per AC; `tdd` writes E2E per criterion; `verify` checks E2E coverage separately |
| 6 | **Observability belongs inside the harness** | Every phase-closing skill appends Observation block with `done-signal` + `verdict-source`; downstream skills read these |
| 7 | **Every session must leave clean state** | `close` mandatory after merge; Phase 0 session-start check in all workflows; `task` warns if ACTIVE_TASK non-empty |

## How to Use

### Starting a task
```
/task
```
Then follow the "Next:" prompt at the end of each skill's output.

### Jumping to a workflow
Pick by task type:
- `/workflow full-sdlc` ŌĆö new feature, complete lifecycle
- `/workflow bug-fix` ŌĆö fixing a bug
- `/workflow feature-build` ŌĆö feature with lighter planning
- `/workflow refactor` ŌĆö structural improvement, no behavior change

### Resuming mid-task
Read `ACTIVE_TASK.md` ŌĆö find the last Observation block. The skill that produced it is the last completed phase. Run the next skill in the workflow sequence.

### Diagnosing a phase result
If a phase outcome seems wrong, check its Observation block's `verdict-source`. If `self-reported` when `external-evidence` is expected, re-run the phase.

## State

All task state lives in `ACTIVE_TASK.md` at repo root. Fixed schema ŌĆö one section per phase:

```
## Requirement        ŌåÉ task
## Design             ŌåÉ design
## ADRs               ŌåÉ grill (locked sentinel required downstream)
## Risks              ŌåÉ risk
## Implementation Log ŌåÉ code, tdd, refactor
## Test Results       ŌåÉ tests, coverage, verify
## Review Findings    ŌåÉ review, audit
## Deploy Checklist   ŌåÉ deploy
## Post-Deploy        ŌåÉ ship
```

Each section may contain one or more `### Observation` blocks written by the skill that populated it.

`ACTIVE_TASK.md` is reset to empty schema by `close` after each merge.

## Phase Gating

Every skill hard-blocks if its required prior section is missing. Observation blocks add a second layer: downstream skills check that the prior phase's Observation exists and shows appropriate evidence, not just that the section is non-empty.

| From phase | To phase | Gate |
|------------|----------|------|
| Intake | Planning | ## Requirement populated |
| Planning | Implementation | ## Design + ## ADRs populated (ADRs locked) + ## Risks Observation `planning-gate: confirmed` |
| Implementation | Testing | ## Implementation Log with test-run-output Observation |
| Testing | Review | ## Test Results PASS with external-evidence Observation |
| Review | Integration | ## Review Findings + all CRITICAL resolved |
| Any | close | ## Review Findings populated (post-merge) |

## Skill Pattern

Every skill follows this pattern:

```
1. Read ACTIVE_TASK.md ŌåÆ required section
2. Hard block if empty
3. Check prior phase Observation block ŌĆö hard block if evidence missing
4. Self-inject: extract relevant fields into meta-prompt
5. Generate output with external evidence where required
6. Write to ACTIVE_TASK.md ŌåÆ own section
7. Append Observation block with done-signal + verdict-source
8. Tell user: "Next: run [next-skill]"
```

## Task Archival

After merge, run `close`. It:
1. Scans all Observation blocks ŌĆö warns if key phases lack evidence
2. Archives full ACTIVE_TASK.md to `task-log/YYYYMMDD-[TYPE]-slug.md`
3. Regenerates `.claude/context/FE_CONTEXT.md` and/or `BE_CONTEXT.md`
4. Resets ACTIVE_TASK.md to empty schema
5. Verifies reset (reads back to confirm)

Type tags: `[FE]` `[BE]` `[FULLSTACK]` `[BUGFIX]` `[REFACTOR]` `[INFRA]` `[DOCS]`

## Forbidden

- Skipping phase order (e.g. `code` before `design`)
- Skipping `verify` before `review`
- Skipping `review` or `audit` before `deploy`
- Skipping `close` after merge (context files become stale, ACTIVE_TASK.md stays dirty)
- Starting a new task without resetting ACTIVE_TASK.md
- Writing multiple tasks into one ACTIVE_TASK.md
- Accepting a `self-reported` Observation verdict where `external-evidence` is required
- Declaring a phase complete without writing its Observation block
- Running `code` without `planning-gate: confirmed` in `## Risks` Observation
- Reading full ACTIVE_TASK.md in any skill that defines section-scoped read boundaries (only `close` may read full file)
- Passing full ACTIVE_TASK.md to `sdlc-reviewer` ŌĆö pass only diff + AC + apiContracts

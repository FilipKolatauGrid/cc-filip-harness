# Claude Code SDLC Harness

## Session Init

At session start, load in order:
1. Read `docs/SKILL_REGISTRY.md` — skill index, phase order, ACTIVE_TASK section map
2. Read `ACTIVE_TASK.md` — current task state (empty = no active task)
3. If `.claude/context/BE_CONTEXT.md` exists → read it (backend codebase snapshot)
4. If `.claude/context/FE_CONTEXT.md` exists → read it (frontend codebase snapshot)

Context files are generated by `close` after each merge. They are the fastest way to understand the codebase without re-scanning source files. Always prefer them over cold file reads when starting a session.

## Meta / DX

- **caveman** ([install](https://github.com/juliusbrussee/caveman)): Reduces output tokens ~75% with full technical accuracy. Install once, active every session.
- **grill-me**: Inspired the `/grill` skill's interrogation pattern. `/grill` implements the logic directly — no separate invocation needed.

## How to Use

### Starting a task
```
/task
```
Then follow the "Next:" prompt at the end of each skill's output.

### Jumping to a workflow
Pick by task type:
- `/workflow full-sdlc` — new feature, complete lifecycle
- `/workflow bug-fix` — fixing a bug
- `/workflow feature-build` — feature with lighter planning
- `/workflow refactor` — structural improvement, no behavior change

### Resuming mid-task
Read `ACTIVE_TASK.md` — find the last populated section, run the skill that reads it next (see `SKILL_REGISTRY.md`).

## State

All task state lives in `ACTIVE_TASK.md` at repo root. Fixed schema — one section per phase:

```
## Requirement        ← /task
## Design             ← /design
## ADRs               ← /grill
## Risks              ← /risk
## Implementation Log ← /code, /tdd, /refactor
## Test Results       ← /tests, /coverage, /verify
## Review Findings    ← /review, /audit
## Deploy Checklist   ← /deploy
## Post-Deploy        ← /ship
```

`ACTIVE_TASK.md` is reset to empty schema by `close` after each merge.

## Phase Gating

Every skill hard-blocks if its required prior section is missing. Gates enforce:

| From phase | To phase | Gate |
|------------|----------|------|
| Intake | Planning | ## Requirement populated |
| Planning | Implementation | ## Design + ## ADRs populated |
| Implementation | Testing | ## Implementation Log populated, TDD green |
| Testing | Review | ## Test Results PASS verdict |
| Review | Integration | ## Review Findings, all CRITICAL resolved |
| Any | /close | ## Review Findings populated (post-merge) |

## Skill Pattern

Every skill follows this pattern — read it as: self-inject → guard → work → write → suggest next.

```
1. Read ACTIVE_TASK.md → required section
2. Hard block if empty
3. Self-inject: extract relevant fields into meta-prompt
4. Generate output
5. Write to ACTIVE_TASK.md → own section
6. Tell user: "Next: run [next-skill]"
```

## Task Archival

After merge, run `close`. It:
1. Archives full ACTIVE_TASK.md to `task-log/YYYYMMDD-[TYPE]-slug.md`
2. Regenerates `.claude/context/FE_CONTEXT.md` and/or `BE_CONTEXT.md`
3. Resets ACTIVE_TASK.md to empty schema

Type tags: `[FE]` `[BE]` `[FULLSTACK]` `[BUGFIX]` `[REFACTOR]` `[INFRA]` `[DOCS]`

## Forbidden

- Skipping phase order (e.g. `code` before `design`)
- Skipping `verify` before `review`
- Skipping `review` or `audit` before `deploy`
- Skipping `close` after merge (context files become stale)
- Starting a new task without resetting ACTIVE_TASK.md
- Writing multiple tasks into one ACTIVE_TASK.md
